# v2 Changes

## Bottom Line

> TL;DR; Easier consumption (and better querying) of market data, and safer trading via enhanced order validation.

- New functions and improved response data formatting for markets (by type: `defaultBinary`, `labelledBinary`, `multiOutcome`)
  - Every outcome is classified into a typed variant with parsed, type-specific fields (underlying, targetPrice, expiry, questionId, isFallback, etc.) and pre-computed side identifiers (coinNum, coin string, asset ID)
  - Raw API response still attached on every market for anything the parsed format does not cover
  - Single `fetchMarkets()` endpoint with type filtering, grouping by type or question, and pagination
- Improved validation for order placement
  - Price tick alignment (5 significant figures) and trailing zero stripping applied automatically before signing
  - Notional check rejects orders below 10 USDH before they hit the exchange
  - Minimum shares validation when mark price is provided
  - Builder fee support with address lowercasing for signing correctness

> No breaking changes. All existing APIs work as before.

## How to Upgrade

### Breaking changes

None. All existing APIs (`fetchEvents`, `fetchEvent`, `fetchCategories`, `placeOrder`, `cancelOrder`) work exactly as before.

Two behavioral changes in `placeOrder` that could surface as new rejections for orders that previously would have been sent to the exchange and rejected there:

1. Limit orders with `price * size < 10 USDH` now return `{ success: false }` before hitting the exchange.
2. Limit order prices are now tick-aligned (5 significant figures) instead of using the old magnitude-bracket rounding. In practice this means more decimal precision, not less, so existing valid prices will still work. But a price like `45.67` that previously rounded to `45.7` now stays as `45.67`.

## Market Type System

All HIP-4 outcomes are now classified into a discriminated union with three types:

- `defaultBinary` -- recurring price binary markets (structured description, Yes/No sides)
- `labelledBinary` -- standalone binary markets with custom side names (e.g. "Hypurr" vs "Usain Bolt")
- `multiOutcome` -- outcomes grouped under a question, with a fallback outcome

Each market carries a `type` field for discrimination, shared fields (`outcomeId`, `name`, `description`, `sides`, `raw`), and type-specific fields where applicable.

### DefaultBinaryMarket

Parsed from the `class:priceBinary|underlying:BTC|expiry:...|targetPrice:...|period:...` description format. Adds `underlying`, `targetPrice`, `expiry` (Date), and `period`.

### LabelledBinaryMarket

Standalone outcomes not in a question group and not parseable as priceBinary. The custom side labels are the primary data.

### MultiOutcomeMarket

Outcomes that belong to a question's `namedOutcomes` or `fallbackOutcome`. Adds `questionId`, `questionName`, `questionDescription`, `isFallback`, and `rawQuestion`.

### Raw response

Every market attaches the original `HLOutcome` on `.raw` so consumers are never locked out of upstream data.

## fetchMarkets API

New method on the events adapter:

```typescript
// All markets, flat array
const markets = await adapter.events.fetchMarkets();

// Filter by type
const defaults = await adapter.events.fetchMarkets({ type: "defaultBinary" });

// Group by type
const grouped = await adapter.events.fetchMarkets({ groupBy: "type" });
// => { defaultBinary: [...], labelledBinary: [...], multiOutcome: [...] }

// Group by question (multiOutcome grouped by questionId, others under "standalone")
const byQuestion = await adapter.events.fetchMarkets({ groupBy: "question" });

// Pagination
const page = await adapter.events.fetchMarkets({ limit: 10, offset: 20 });
```

Cached at the outcomeMeta level with the same 30s TTL as the existing events cache.

## Pricing Utilities

New module at `src/adapter/hyperliquid/pricing.ts`:

- `computeTickSize(price)` -- 5-significant-figure tick size per the exchange spec: `10^(floor(log10(price)) - 4)`
- `roundToTick(price)` -- round to nearest valid tick
- `formatPrice(price)` -- round to tick, then strip trailing zeros (critical for signing correctness)
- `stripZeros(s)` -- strip trailing zeros from any numeric string
- `getMinShares(markPx)` -- minimum order size in shares, solving `size * min(markPx, 1 - markPx) >= 10 USDH`
- `MIN_NOTIONAL` -- constant, 10 USDH

### Why trailing zeros matter

The exchange strips trailing zeros before msgpack hashing. Sending `"0.650"` when the server hashes `"0.65"` produces a different hash, which causes wrong signer recovery. All price and size values in signed payloads must go through `formatPrice()` or `stripZeros()`.

## Order Validation

`placeOrder()` now validates before signing and submitting:

- **Notional check**: rejects if `price * size < 10 USDH` for limit orders
- **Minimum shares check**: when `markPx` is provided, rejects if `size < getMinShares(markPx)`
- **Price tick alignment**: limit order prices are rounded to the correct tick size and stripped of trailing zeros
- **Size formatting**: trailing zeros stripped from the size string before signing

These return `{ success: false, error: "..." }` rather than hitting the exchange and getting a cryptic rejection.

## Builder Fee Support

`PredictionOrderParams` now accepts:

- `builderAddress` -- referral address (lowercased before signing, since checksummed addresses cause wrong signer recovery)
- `builderFee` -- fee in tenths of a basis point (0 = none, 100 = 0.1%, 1000 = 1.0% max)

When both are provided and `builderFee > 0`, the `builder` field is included in the signed order action.

## Market Discovery

New module at `src/adapter/hyperliquid/market-discovery.ts`:

- `parseDescription(desc)` -- parses the pipe-delimited `class:priceBinary|...` format into a typed `ParsedDescription`
- `discoverPriceBinaryMarkets(meta, mids)` -- filters to active priceBinary markets with future expiry and live underlying price
- `timeToExpiry(market)` -- minutes until expiry (negative = expired)
- `periodMinutes(period)` -- parse "15m", "1h", "1d" to minutes
- `formatMarketLabel(market)` -- "BTC-1d", "HYPE-15m"

These are standalone utilities. For the integrated approach, use `fetchMarkets({ type: "defaultBinary" })` instead.

## Classification Engine

`classifyOutcome(outcome, questions)` and `classifyAllOutcomes(outcomes, questions)` implement the detection logic:

1. Description parses as `class:priceBinary` -> `defaultBinary`
2. Outcome is in a question's `namedOutcomes` or `fallbackOutcome` -> `multiOutcome`
3. Everything else -> `labelledBinary`

The integration test suite verifies every testnet outcome classifies into a known type and will flag any unclassified outcome as `POTENTIAL NEW MARKET TYPE - INVESTIGATE` with the full raw response.

## Updated PredictionOrderParams

New optional fields on the trading params type:

```typescript
interface PredictionOrderParams {
  // ... existing fields ...
  markPx?: number; // mark price for min-shares validation
  builderAddress?: string; // referral address (will be lowercased)
  builderFee?: number; // tenths of a basis point (0-1000)
}
```
